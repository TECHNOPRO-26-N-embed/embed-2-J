## 1. システム全体像

### 1-1. ブロック構成（入力 → Arduino → 出力）

```
【入力】        【Arduino UNO R3】       【出力】
プッシュスイッチ ─→ D2 ──┐                           ┌→ D3 ─→ 受動ブザー
                         │   判定・制御・監視まとめ  ─┤
                         └─────────────────────────┘
                                     │
                                     └→ D4 ─→ LED（任意）
```
補足：  
- D2ピンはINPUT_PULLUPでボタン検知  
- D3ピンでtone()利用・アクティブブザー出力  
- D4はLED出力（任意。設置しない場合は未接続）

---

### 1-2. 状態遷移（ガジェットの「ふるまい」設計）

```
[電源ON/初期化]
  ↓（初期化完了）
[待機]──（ボタン押下）──→[警報発報]
  ↑  └────────────（ボタン解除/タイムアウト/長押し終了）──┘
```

**各状態の意味:**

| 状態名      | この状態のとき何をしているか                                  | LED・ブザーの様子             |
|:-----------|:------------------------------------------------------------|:------------------------------|
| 初期化      | ピン初期化／変数リセット                                    | 消灯、ブザー停止              |
| 待機        | ボタン状態監視(チャタリング対策有)、入力待ち                 | 消灯、ブザー停止              |
| 警報発報    | ボタン押下中：ブザー鳴動・LED点灯（任意）                   | ブザーON、LED点灯             |
| 自動停止    | 長押し状態で一定時間後自動OFF（追加機能）                   | OFF                           |

## 2. ソフトウェア設計

### 2-1. データ設計（使う変数・定数の洗い出し）

| 情報の名前     | 何を表すか                          | 型         | バイト数 | 初期値 | 備考                  |
|:--------------|:----------------------------------|:-----------|:--------|:------|:----------------------|
| buttonState   | ボタン現在状態(LOW＝押下)           | bool       | 1B      | HIGH  | digitalRead結果       |
| lastButtonState| 前回のボタン状態                   | bool       | 1B      | HIGH  | チャタリング判定用    |
| lastDebounceTime| 最終変化検出時刻                  | unsigned long| 4B     | 0     | チャタリング対策用    |
| debounceDelay | チャタリング許容時間[ms]            | const int  | 2B      | 50    | 50ms等                |
| alarmState    | 現在のシステム状態 enum:WAIT/ON     | int        | 2B      | 0/WAIT|                       |
| ledState      | LED点灯状態（追加機能用）           | bool       | 1B      | LOW   | 任意                  |
| buzzerFreq    | ブザー出力周波数                    | int        | 2B      | 523   |                        |
| holdStartTime | 長押し判定開始時刻                  | unsigned long | 4B    | 0     | 長押し追加機能        |

> **SRAM計算例：合計 約17B**  
> Arduino UNO R3は2048B。十分余裕あり。

---

### 2-2. 機能一覧・関数一覧（役割の定義）

| 機能ID | 機能名          | 関数名             | 担う「1つの仕事」                       | 主な引数 | 戻り値 | 呼出し元   |
|:------|:----------------|:-------------------|:-----------------------------------------|:-------|:-----|:-----------|
| —     | 初期化          | setup()            | I/Oピン初期化、変数初期化               | なし    | なし  | 起動時1回  |
| —     | メイン制御      | loop()             | 状態監視→各種判定関数呼び出し           | なし    | なし  | 常時       |
| F01   | ボタン入力検知  | readButton()       | チャタリング済みの確定入力を返す         | なし    | bool  | loop,他    |
| F02   | ブザー出力      | setBuzzer(onoff)   | 指定状態でtone()/noTone()制御            | onoff   | なし  | 状態更新時 |
| F03   | LED点灯制御(任意)| setLED(onoff)      | 指定状態でLED点灯制御                    | onoff   | なし  | 状態更新時 |
| F04   | 長押し判定(任意) | checkLongPress()   | 所定時間保持で“長押し”を検出             | なし    | bool  | 状態変化時 |
| F05   | 自動停止(任意)  | checkTimeout()     | ボタン一定時間押しっぱなし→自動停止      | なし    | bool  | loop,他    |

### 2-3. タイミング設計（delay vs millis）

| 処理           | 必要な周期 | delay/millis | 判断理由                         |
|:---------------|:----------|:------------|:---------------------------------|
| ボタン監視     | 10ms      | millis      | 常時監視・他処理と並行           |
| LED点灯制御    | 状態変化時 | millis      | 他処理と並行                     |
| ブザー鳴動     | 状態変化時 | millis      | 他処理と並行                     |
| 長押し判定     | 10ms      | millis      | ノンブロッキングで判定           |
| （delayは使わない）|          |            |                                 |

### 3-2. 電気的考慮事項

| 用途                     | 判断・計算メモ                                       |
|:------------------------|:---------------------------------------------------|
| LED電流制限抵抗          | (5V−2V)÷0.02A=150Ω → 安全をみて220Ω                |
| ボタンのプルアップ       | INPUT_PULLUP利用／外付け抵抗不要                    |
| ブザー動作電圧           | 5V／tone()利用可能対象                              |

## 4. 異常系・エラー処理の方針

| 異常ケース   | 発生条件             | 検知方法                     | 対応方針                         |
|:------------|:--------------------|:-----------------------------|:---------------------------------|
| チャタリング| ボタン押下時         | 前回変化時から50ms未満        | debounce(デバウンス)で入力無視   |
| ノイズ      | アナログ入力揺れ     | 閾値バッファ設定              | バッファ以内変化は保持           |
| ブザー異常   | tone/noTone効かない  | 実行後に出力状態確認          | 電気的故障時は再起動促進         |
| LED断線     | LED点かない          | 実出力確認                    | 部品交換等物理修理               |

## 7. AIレビュー記録

### Q1: 設計の妥当性確認

> 「この基本設計書をもとにArduino UNO R3でコードを書きます。ピン競合・ライブラリ競合・状態遷移の抜け・delay()の問題点はありますか？」

**AIの回答（要約）：**  
- ピン（D2/D3/D4）は競合なし。I2C・SPIともバッティング無し。D0/D1は未使用でOK。
- tone/noTone用ライブラリは標準。追加インクルードなし。
- 状態遷移上の抜け無し。各状態から待機状態に必ず戻れる設計。
- delayは使わず、millisベースで非同期処理を設計している。問題なし。

**対応した内容：**  
- すべて対応済み。

### Q2: チャタリング・delay・ノイズの確認

> 「この設計にチャタリング対策は含まれていますか？delay()を使っている箇所で問題はありますか？アナログセンサーを使う場合、ノイズ対策は必要ですか？」

**AIの回答（要約）：**  
- チャタリング対策は変化時刻＋50ms判定で対応。
- delay処理は用いず、ノンブロッキングで設計済み。
- アナログセンサー非使用だが、今後応用時は閾値バッファを推奨。

**対応した内容：**  
- デバウンス用時刻管理・閾値バッファ設計を明記。

### Q3: 要件への対応漏れ確認（任意）

> トレーサビリティ表とSection 5で設計・テストの紐付けチェック

**AIの回答（要約）：**

**対応した内容：**
