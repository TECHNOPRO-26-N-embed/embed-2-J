### `setup()` — 初期化処理
  【処理の流れ】

1. Serial.begin(9600) を実行する

2. ボタンを INPUT_PULLUP に設定する

3. LED とブザーを OUTPUT に設定する

4. SPI.begin() を実行する

5. RC522 RFIDモジュールを初期化する

6. 起動確認として LED を短時間点灯する



### `loop()` — メインループ

【処理の流れ】

＜毎ループ実行すること＞
  - ボタン状態を確認する

＜currentState が 0（待機中）のとき＞
  - ボタン押下を待つ
  - ボタンが押されたら currentState = 1

＜currentState が 1（認証中）のとき＞
  - RFIDカードを読み取る
  - UIDを比較する
  - UIDが一致した場合 currentState = 2
  - UIDが不一致の場合 currentState = 3

＜currentState が 2（認証成功）のとき＞
  - LEDを点灯する
  - 一定時間後に待機状態へ戻る

＜currentState が 3（警告中）のとき＞
  - ブザーを鳴らす
  - 一定時間後に待機状態へ戻る


  ### readButton() — ボタン状態を読み取る

  【処理の流れ】

1. digitalRead() でボタン状態を取得する

2. LOWなら押下と判定する

3. チャタリング防止処理を行う

4. ボタン押下結果を返す

【エラー・異常ケース】

- ノイズによる誤検知は50ms以内を無視する

### readRFID() — RFIDカードを読み取る
【処理の流れ】

1. カードが存在するか確認する

2. UIDを読み取る

3. UID文字列を生成する

4. UIDを返す

【エラー・異常ケース】

- カードが存在しない場合は読み取りを行わない

### checkUID() — UIDを比較する
【処理の流れ】

1. 読み取ったUIDと登録済みUIDを比較する

2. 一致した場合 true を返す

3. 不一致の場合 false を返す

【エラー・異常ケース】

- UIDが空の場合は false を返す

### updateLED() — LED制御を行う
【処理の流れ】

1. 認証成功時にLEDを点灯する

2. 一定時間後に消灯する

### updateBuzzer() — ブザー制御を行う
【処理の流れ】

1. 認証失敗時にブザーを鳴らす

2. 一定時間後に停止する

## 3. 重要ロジックの詳細設計

### 3-1. チャタリング防止（デバウンス処理）

【考え方】
  ボタンが押されたとき、50ms以内の連続入力は同じ1回の押下として扱う。

【処理の流れ】
1. ボタン入力を取得する
2. 前回入力時刻との差を確認する
3. 50ms未満の場合は無視する
4. 50ms以上の場合のみ押下として確定する

【必要な変数】
  lastDebounceTime : unsigned long
  DEBOUNCE_DELAY : const int = 50

  ### 3-2. millis() を使ったタイマー管理
  【考え方】
  delay() を使わず、millis() を利用して並行処理を行う。

【処理の流れ】
1. now = millis() を取得する
2. 現在時刻と前回時刻との差を確認する
3. 一定時間経過後にLEDやブザーを停止する

【millis() を使う処理】
  ・認証成功時のLED点灯時間
  ・ブザー警告時間

  ### 3-3. UID比較ロジック
  【処理の流れ】

1. RFIDカードのUIDを取得する

2. 登録済みUIDと比較する

3. 一致した場合：
   認証成功状態へ変更する

4. 不一致の場合：
   警告状態へ変更する

【入力値と出力値の関係】

- 正しいUID → LED点灯
- 間違ったUID → ブザー警告

## 6. AIレビュー記録

### Q1: 実装上の問題確認
AIの回答（要約）
delay() を多用するとボタン入力やRFID読み取りが停止する可能性がある。
RFIDのUID比較時に文字列形式を統一する必要がある。
対応した内容
millis() を利用する設計へ変更した。
UID比較形式を統一した。

### Q2: 単体テスト仕様の確認
AIの回答（要約）
RFID未検知時や連続入力時のテストを追加すると良い。
対応した内容
入力系テストへ追加した。
